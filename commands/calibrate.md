# Command: calibrate

## 用法

```
/calibrate <目录> [--level <production|good|prototype>]
```

**参数说明**：
- `<目录>`：已知质量水平的代码目录，用于校准评分规则
- `--level`：该代码库的预期质量水平（可选）
  - `production`：生产就绪级别，期望总分 54–60，大多数维度 >= 8
  - `good`：良好工程质量，期望总分 42–53，大多数维度 >= 6
  - `prototype`：原型/快速开发级别，期望总分 30–41
  - 若未指定，AI 自动推断并请用户确认

**示例**：
```
/calibrate ./flask --level production
/calibrate ./my-library --level good
/calibrate ./my-library
```

**适用场景**：
- 已知高质量库的得分异常低
- 怀疑某个维度的规则过严或过松
- 想用真实代码对行业基准进行校准

---

## 执行流程

### 阶段 0：参数校验

统计非 `--level` 的参数：
- 1 个目录路径：正常
- 0 个或 2 个以上目录：报错停止

```
用法：
  /calibrate <目录>                          # AI 自动判断预期质量水平
  /calibrate <目录> --level production       # 手动指定：生产就绪（54–60 分）
  /calibrate <目录> --level good             # 手动指定：良好工程质量（42–53 分）
  /calibrate <目录> --level prototype        # 手动指定：原型级别（30–41 分）
```

校验规则：
- `<目录>` 必须存在且包含至少 1 个 .py 文件（calibrate 需要可扫描的 Python 代码）
- `--level` 若指定，值必须是 `production`、`good`、`prototype` 之一，否则报错停止

记录 `DIR` 和 `LEVEL`（可能为空），供后续阶段使用。

---

### 阶段 1：环境检查与安装

运行 `hooks/auto-install.sh`。

若有工具安装失败，询问用户：
```
工具 <名称> 安装失败，是否继续（跳过对应维度的校准）？[y/n]
```

---

### 阶段 2：数据采集与评分

**第一步：调用 scanner agent**

调用 `agents/scanner.md`，传入：
- `MODE=single`
- `DIR=<目录路径>`

等待 `SCANNER_DONE` 信号。若 scanner 报错，输出错误信息并停止。

**第二步：调用 interpreter agent**

调用 `agents/interpreter.md`，传入 scanner 的 JSON 输出，使用 single 模式。

**本次为 calibrate 场景的特殊说明**：请 interpreter 在输出 JSON 的 `security` 字段中额外附带 `bandit_issues_raw` 数组，包含每个问题的以下字段：
- `test_id`：bandit 规则 ID（如 B324）
- `issue_text`：告警原始描述文本
- `severity`：HIGH / MEDIUM / LOW
- `confidence`：HIGH / MEDIUM / LOW

这些原始数据用于后续诊断阶段区分"误报"和"真实风险"，不影响评分逻辑本身。

等待 `INTERPRETER_DONE` 信号。

---

### 阶段 3：确定预期质量水平

**若 `--level` 已指定**，直接使用对应区间，跳过本阶段。

**若 `--level` 未指定**，执行以下步骤：

1. 读取目录中的 `README.md`（若存在）
2. 读取 2–3 个有代表性的 .py 文件（优先选测试文件、核心模块）
3. 综合以下信号判断预期质量水平：
   - 项目类型（库/框架/应用/脚本）
   - 是否有完整测试套件
   - 代码注释质量和文档完整性
   - 是否有类型注解、CI/CD 配置等工程化标志

输出推断结果并请用户确认：

```
────────────────────────────────────────
自动判断代码预期质量水平

  分析目录：<DIR>
  推断水平：<production / good / prototype>
  推断依据：<一句话说明>

  对应期望区间：
    总分：<XX–XX> 分
    各维度：大多数维度应 >= <X> 分

  是否确认？
  输入 y 确认 / 输入 production / good / prototype 修正
────────────────────────────────────────
```

根据用户响应确定最终 `LEVEL`。

期望区间对照表：

| LEVEL | 期望总分区间 | 各维度期望下限 |
|-------|------------|--------------|
| production | 54–60 | >= 8（允许 1 个维度 >= 6） |
| good | 42–53 | >= 6（允许 1–2 个维度 >= 4） |
| prototype | 30–41 | 无特定维度下限 |

---

### 阶段 4：诊断分析

读取 `standards/references.md`，结合实际评分 JSON 和期望区间，执行诊断。

**第一步：计算各维度偏差**

对每个维度，计算：偏差 = 实际分 - 该级别期望下限分。

偏差为负表示"低于预期"。重点诊断偏差 <= -2 的维度。

**第二步：逐维度判断原因**

对每个偏差较大的维度，严格区分以下两类原因，并给出具体证据：

**A. 规则过严（建议修改 references.md）**：
- **安全维度**：`bandit_issues_raw` 中的告警 `issue_text` 描述的是框架或语言的惯用法，而非真实漏洞。判断关键词：
  - 倾向误报：`bind`, `debug mode`, `jinja2 autoescape`, `ssl_verify`, `tmp file`
  - 结合上下文：exec/subprocess 用于开发工具脚本，SHA/MD5 用于非安全场景（如文件 hash）
- **简洁性维度**：jscpd 标记的重复块属于测试 fixture、配置模板、import 声明等样板代码（boilerplate）
- **复杂度维度**：radon 评级为 B（8 分）而预期下限也是 8 分，实际无偏差，不属于"规则过严"

**B. 代码本身有问题（不修改规则）**：
- **安全维度**：告警描述的是真实风险。判断关键词：
  - `sql`, `injection`, `pickle`, `deserializ`, `hardcoded`, `secret`, `password`, `weak hash`, `sha1 for security`, `md5 for security`
- **简洁性维度**：重复的是业务逻辑、算法实现或数据处理代码
- **其他维度**：客观工具数据不佳（高复杂度、高 bug 密度），且验证后属实

**第三步：输出诊断报告**

```

  校准诊断报告
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  重点维度分析（偏差 >= 2 分的维度）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  - <test_id>（<severity>/<confidence>，-X 分）：<issue_text> — <为什么属于误报/惯用法>
```

### 阶段 5：提议规则修改

**若所有偏差均来自代码本身的问题**，输出以下内容并结束流程：

```
诊断结论：该代码库的得分偏低，原因是代码本身存在以上问题，
评分规则本身并无明显偏差。不建议修改 standards/references.md。

如需改善得分，建议修复以上代码问题后重新运行 /quality-check。
```

---

**若存在"规则过严"的情况**，生成修改提议。

**重要约束**：提议必须是对**评分规则本身的改进**（调整阈值、权重、评分机制等），不允许写"对特定项目/框架的豁免"。规则应保持通用性，对所有代码库公平适用。

按以下格式展示每条提议：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  规则修改提议
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  共 <N> 处修改建议，预计将总分提升约 <+X> 分。

  ────────────────
  提议 <编号>：<维度名> — <修改的简短描述>
  ────────────────

  修改位置：standards/references.md > <章节标题>

  修改原因：
  <一到两句话，说明为什么当前规则过严，以及修改后的规则更合理>

  BEFORE：
  ┌──────────────────────────────────────────
  │ <原始文本，逐字引用，不省略>
  └──────────────────────────────────────────

  AFTER：
  ┌──────────────────────────────────────────
  │ <修改后的完整文本>
  └──────────────────────────────────────────

  不建议修改的项目（代码本身有问题，修规则会掩盖真实风险）：
  - <test_id>（<severity>）：<一句话说明为何应修代码而非修规则>

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  是否确认应用以上修改？

  输入 y      确认全部应用
  输入 n      取消，不修改任何规则
  输入 1,3    仅应用编号 1 和 3 的提议（逗号分隔）

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

---

### 阶段 6：应用修改

**若用户输入 `n`**：

```
已取消。standards/references.md 未做任何修改。

你可以重新运行 /calibrate 或直接编辑 standards/references.md 进行手动调整。
```

结束流程。

---

**若用户输入 `y` 或数字序列（如 `1,3`）**：

1. 确定需要应用的提议编号列表
2. 读取当前 `standards/references.md` 全文
3. 逐条应用确认的修改，不改动文件中其他内容
4. 在文件末尾的版本记录表追加一行：

```
| v<下一个版本号> | <本次修改的一句话摘要>（基于 <DIR> 校准，共 <N> 处修改） |
```

5. 将完整更新后的文件写回 `standards/references.md`

6. 输出确认信息：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  standards/references.md 已更新

  应用的修改：<N> 处
  跳过的修改：<M> 处（用户未确认）
  版本记录：已追加

  建议：运行 /quality-check <DIR> 验证新规则效果。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 错误码参考

| 情况 | 行为 |
|------|------|
| 目录不存在 | 报错并停止，提示检查路径 |
| 目录无 .py 文件 | 报错并停止：calibrate 需要可扫描的 Python 代码 |
| `--level` 值非法 | 报错并提示合法值 |
| scanner 报错 | 输出错误并停止 |
| 所有维度均符合预期 | 输出"评分系统与该代码库吻合，无需调整"并结束 |
| `standards/references.md` 写入失败 | 报错，建议用户检查文件权限 |
